## Task

Implement authentication for the Project DNA CLI tool. Given that Project DNA is a local-first CLI application that relies on an LLM provider abstraction layer (currently Fireworks AI via the OpenAI SDK), authentication in this context most plausibly refers to API key management, credential validation, and secure provider configuration for the AI reasoning engine.

The implementation should cover:
1. API key retrieval from environment variables (leveraging the existing `dotenv` dependency).
2. Validation of API key presence and format before attempting LLM calls.
3. A CLI command or initialization step that allows users to configure or verify their provider credentials.
4. Integration with the existing provider abstraction layer so that authentication is decoupled from any single LLM provider.
5. Clear error handling and user-facing messages (using `chalk` and `ora` for CLI UX consistency) when authentication fails or credentials are missing.

## Relevant Project Context

Project DNA is an AI-powered architectural governance platform that operates as a CLI tool. It preserves technical and business knowledge of software projects in a local `.pdna` directory. The system uses a provider abstraction layer to communicate with LLM providers (currently Fireworks AI) for reasoning and knowledge extraction. Authentication is explicitly listed as missing context in the architecture insights, meaning there is no existing authentication mechanism to extend or refactor.

## Relevant Domains

- **AI Integration**: The provider abstraction layer is the primary consumer of authenticated credentials. Authentication must integrate seamlessly with this layer.
- **CLI**: Users interact with Project DNA exclusively through the command line. Credential configuration and validation must be exposed via CLI commands.
- **Local Storage**: The local-first architecture implies that any persisted configuration (such as provider selection or credential metadata) should reside in the `.pdna` directory or user-level configuration, not in external databases.
- **Schema Validation**: If credentials or provider configuration are persisted, they should be validated against a predefined schema using `zod`.

## Architecture Notes

- The architecture is local-first: no external databases or remote authentication services are expected.
- Provider abstraction is a core architectural rule. Authentication logic must not be tightly coupled to a specific provider (e.g., Fireworks AI or OpenAI). It should be designed to support multiple providers.
- The system currently relies on Fireworks AI for reasoning, which is identified as a medium-impact provider dependency risk. Authentication should be extensible to accommodate additional providers.
- The `.pdna` directory is the canonical location for local persistence. If credential metadata (not raw secrets) needs to be stored, it should follow this convention.
- Raw API keys should not be persisted in plaintext files. Prefer environment variables loaded via `dotenv`.

## Technologies / Dependencies

- **TypeScript**: All new implementation must be in TypeScript.
- `dotenv`: Already a dependency; use for loading environment variables including API keys.
- `openai`: The SDK used for LLM provider communication; authentication credentials will be passed to this client.
- `commander`: CLI framework for defining commands related to auth configuration.
- `chalk` and `ora`: Use for consistent CLI output formatting and spinner states during auth validation.
- `zod`: Use for schema validation of any configuration objects.
- `fs-extra`: Use for file operations if credential metadata needs to be read from or written to the `.pdna` directory.

## Coding Constraints

- Prefer TypeScript for all new implementation work.
- Maintain provider abstraction: authentication must be provider-agnostic at the interface level.
- Keep architecture knowledge centralized in `.pdna` files; do not introduce external storage.
- Ensure all AI-generated or persisted output is validated against predefined schemas before persistence.
- Do not hardcode API keys, provider URLs, or environment variable names in a way that breaks provider abstraction.
- Follow existing CLI command patterns established by the `commander`-based command structure.

## API / Structural Constraints

- No explicit API conventions are defined in the project artifacts. Follow existing code structure within the `src` directory.
- If a provider auth interface is created, it should define a contract for credential retrieval and validation that any LLM provider implementation can satisfy.
- CLI auth commands should follow the naming and structure conventions of existing CLI commands in the project.

## Expected Outcome

- A TypeScript module or set of modules within `src` that handles API key retrieval from environment variables, validation, and error reporting.
- A CLI command (e.g., `pdna auth` or `pdna config`) that allows users to check, set, or validate their provider credentials.
- Integration with the existing provider abstraction layer so that authenticated credentials are passed to the LLM client at runtime.
- Clear, user-friendly CLI messages when credentials are missing, invalid, or expired.
- Schema-validated configuration objects if any credential metadata is persisted locally.
- No hardcoded secrets; all sensitive values must come from environment variables.
